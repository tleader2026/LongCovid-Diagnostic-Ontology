# Project Instructions: Human Regulatory State Engine

This project models post-viral / neuroimmune illness as a dynamic regulatory-state system, not as a fixed diagnosis engine.

Core rule:
Symptoms are observable state indicators. Functional domains, cell populations, and mechanisms are probabilistic hypothesis layers.

Never describe candidate mechanisms as confirmed pathology.

Use this chain:
Symptom -> Trigger -> Timing -> Response Pattern -> Functional Domain -> Candidate Mechanism -> Phenotype -> Pathway -> Outcome

Prioritize:
- explainability
- editable ontology data
- weighted scoring rules
- longitudinal tracking
- careful medical language
- no certainty claims
- local-first operation
- clean admin workflows

The product goal is adaptive phenotype stratification, not diagnosis.

Build order:
v0: ontology schema and seed data
v1: adaptive intake and scoring engine
v2: patient result dashboard
v3: admin ontology editor
v4: longitudinal outcomes tracking
v5: ML-assisted phenotype clustering

Start with v1, but create the schema in a way that supports future longitudinal tracking.

Implementation priorities:
1. Get the database schema right.
2. Seed the ontology cleanly.
3. Build the adaptive intake.
4. Build the scoring engine.
5. Build explainable results.
6. Add admin editing.
7. Polish UI.
